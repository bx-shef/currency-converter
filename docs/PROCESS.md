# Процесс работы

> Last reviewed: 2026-07-23

Как в этом репозитории делается изменение — от ветки до мержа. Конвенции **кода** (где живут
чистые функции, язык комментариев и т.п.) — в [`CLAUDE.md`](../CLAUDE.md) › Конвенции; здесь —
**рабочий процесс**.

## 1. Цикл изменения

```
ветка → правка → pnpm check (зелёный) → PR → проверяющие → устранить замечания → зелёный CI → мерж
```

1. **Ветка** от `main` (не коммитить в `main` напрямую). Понятные коммит-сообщения.
2. **Перед пушем — `pnpm check`** (алиас `lint && typecheck && test`). Те же проверки гоняет CI,
   так что красный `check` = красный CI. Чистая логика — в `app/utils/*` (+ конфиг), покрыта
   юнит-тестами; composables/компоненты — в проекте `nuxt` (`tests/nuxt/`).
3. **PR** в `main`. Тело описывает проблему и что сделано; ссылки на issue (`Closes #N`).
4. **Проверяющие** (см. §2), затем **мерж только по зелёному CI** (см. §3).

## 2. Ревью проверяющими

Каждый PR проверяют **несколько независимых ревьюеров** (для app-кода — до 5: корректность,
тесты, безопасность/приватность, архитектура, тех-директор; для доков — 2–3: точность vs код,
полнота/тех-директор). Практики:

- **Adversarial**: ревьюер пытается найти дыру, а не подтвердить. Замечания делятся на
  **БЛОКЕРЫ / ЗАМЕЧАНИЯ / ЧИСТО**.
- **Формат находки**: кто · что · почему · как исправить.
- **Устранение**: блокеры — обязательно; замечания — по существу (устранить или обосновать
  отказ). После правок — перепроверка.
- Цель — не «зелёная галочка», а поймать то, что автотесты и один проход не ловят (напр. фикс
  #135 оказался в родительском layout, а не там, где правил автор — поймали именно ревьюеры).

## 3. CI и мерж

- **CI** (`.github/workflows/ci.yml`): job `ci` (install → `nuxi prepare` → lint → test →
  typecheck → `nuxt generate`) + `docker-build` на PR (сборка прод-образа без push).
  На `push: main` — job `deploy` (пуш образа в GHCR, Watchtower подхватывает ~5 мин).
- **Красный `ci` блокирует деплой** (образ не попадёт в GHCR). Красный CI на PR — чинишь и
  пушишь снова, тот же цикл, без сокращений.
- **Мерж — только когда `ci` и `docker-build` зелёные.** Метод — squash, заголовок с номером PR.
  Решение о мерже — за владельцем: агент доводит PR до зелёного CI и мержит только по явному
  разрешению (при первичной настройке деплоя мерж всегда за человеком — см. `AI_DEPLOY_GUIDE.md`).
- Отдельного hotfix-пути нет: прод-инцидент идёт по тому же циклу; откат — [`OPERATIONS.md`](OPERATIONS.md).
- Деплой/эксплуатация — [`OPERATIONS.md`](OPERATIONS.md); первичная настройка —
  [`AI_DEPLOY_GUIDE.md`](AI_DEPLOY_GUIDE.md).

## 4. Штамп ревью документов

Каждый `.md` в корне и `docs/` несёт строку `> Last reviewed: YYYY-MM-DD` (ISO-дата)
блок-цитатой **сразу под заголовком H1**. Ключ `Last reviewed` — **всегда на английском**
(технический маркер, ищется одним grep). Дату бампим **только** при содержательном изменении
или проверке актуальности, не при косметике. Наличие штампа во всех отслеживаемых `.md`
проверяет `tests/mdReviewStamp.test.ts` (`reporting-kit/` исключён — свой процесс).

Позиция «сразу под H1» — **соглашение**: тест проверяет только **наличие** штампа (regex с флагом
`m`), не его положение в файле. Держим под H1 ради единообразия.

## 5. Конвенции документации

- **Язык** (канон — [`CLAUDE.md`](../CLAUDE.md) › Конвенции): пользовательский текст и `.md` — рус.,
  комментарии/JSDoc в коде — англ.
- **Без дублирования**: один источник правды, между доками — перекрёстные ссылки, а не копипаста.
- **UI-правки** сверяются с документацией Bitrix24 UI (`b24ui`).
- **Визуальная проверка** UI — скриншот-харнесс `pnpm screenshots` ([`VISUAL_VERIFICATION.md`](VISUAL_VERIFICATION.md)).
- **Приватность телеметрии** — инвариант «shape/outcome, never content» (детали и список целей —
  [`DATA_POLICY.md`](DATA_POLICY.md)).
- Что нельзя покрыть автотестами (реальный портал Б24, живые цели Метрики) — ручной
  [`MANUAL_TEST_PLAN.md`](MANUAL_TEST_PLAN.md).
